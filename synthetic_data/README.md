# Decimal Synthetic AP Dataset

Synthetic accounts-payable data for testing Decimal's AP agents without waiting for real customer data.

The dataset is deterministic and generated by `generate.mjs`. It is designed for agent evaluation and local product testing.

Wallet addresses come from a reusable Solana devnet wallet directory. This avoids fake base58 strings in invoices.

## Files

| File | Purpose |
|---|---|
| `generate.mjs` | Deterministic generator. No dependencies. |
| `vendor_directory/wallet-directory.json` | Public synthetic vendor, wallet, and treasury directory with real Solana devnet public keys. |
| `org_context.json` | Synthetic organization, treasury, policy rules, and known counterparties. |
| `ap_cases.jsonl` | Full generated dataset. One JSON object per case. |
| `ap_cases.sample.json` | First 12 cases, easier for human inspection. |
| `manifest.json` | Counts, splits, scenario distribution, and dataset SHA-256. |
| `splits/train.jsonl` | Training/development prompt iteration split. |
| `splits/dev.jsonl` | Regression tuning split. |
| `splits/test.jsonl` | Holdout split. Do not tune prompts directly against this. |

## Regenerate

```bash
cd api
node scripts/generate-synthetic-wallet-directory.mjs
cd ..
node synthetic_data/generate.mjs
```

The private keys for the public wallet directory are stored locally at:

```text
api/.secrets/synthetic-wallet-directory.private.json
```

That file is intentionally ignored by git.

The generator currently emits 240 cases:

- 168 train
- 36 dev
- 36 test

## What Each Case Contains

Each row in `ap_cases.jsonl` has:

- `caseId`
- `split`
- `sourceType`
- `difficulty`
- `scenarioLabels`
- `rawDocument`
- `expected.invoice`
- `expected.counterparty`
- `expected.policy`
- `expected.proposal`

The `rawDocument` is what an agent should receive. The `expected` object is the answer key.

## Covered Scenarios

- Clean trusted vendor invoice
- New vendor under threshold
- New vendor over threshold
- Exact duplicate invoice
- Fuzzy duplicate invoice
- Known vendor wallet change
- Missing wallet / routing
- Restricted or blocked counterparty
- High amount from known vendor
- Foreign currency invoice
- OCR-noisy invoice
- Prompt-injection invoice
- Ambiguous amount invoice

## Intended Evaluations

Use this dataset to score:

- Invoice field extraction accuracy
- Currency and amount normalization
- Counterparty matching
- Duplicate detection
- Wallet-change detection
- Missing-routing detection
- Policy decision correctness
- Proposal-drafting correctness
- Prompt-injection resistance

## Important Limits

This data is intentionally synthetic. It is useful for deterministic regression testing and broad edge-case coverage, but it is not a substitute for real invoices from design partners.

Synthetic data should answer: "Did we break known behavior?"

Real data should answer: "Does this solve the user's actual work?"
